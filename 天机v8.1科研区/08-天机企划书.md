## 零、架构决策：单进程内核 + 分布式就绪

### 0.1 决策结论

```
天机v10.0.1 = 单进程Memory OS内核（轻量、嵌入、零运维）
           + Protocol接口实现分布式就绪
           + 灵境提供分布式编排能力（混沌，未规划）
```

### 0.2 为什么天机内核必须是单进程

| 理由 | 说明 |
|------|------|
| **嵌入性** | Memory OS必须像SQLite一样可嵌入任何AI平台，单进程=零运维 |
| **开发效率** | 单进程调试简单，模块逐个跑通才可行 |
| **DeepSeek三循环** | 快速反应环<100ms，跨进程通信无法保证延迟 |
| **SQLite优势** | 零配置、零运维、ACID事务、FTS5全文搜索 |
| **MCP协议** | MCP是进程内通信，单进程内调用零序列化开销 |
| **部署简洁** | ONEDIR单EXE，双击即用，无Docker/K8s依赖 |

### 0.3 为什么必须分布式就绪

| 理由 | 说明 |
|------|------|
| **企业级需求** | 多租户、高可用、水平扩展 |
| **知识图谱** | Neo4j/图数据库需要独立进程 |
| **跨平台同步** | 通用记忆层需要多节点协调 |
| **记忆容量** | 单机SQLite上限~280TB，企业需要PB级 |

### 0.4 "分布式就绪"的Protocol接口设计

同一接口，本地/远程两种实现，切换零代码改动：

```python
# 单进程模式(默认，零配置)
storage: IStorageEngine = SQLiteStorage(db_path="data/icme.db")

# 分布式模式(灵境编排，替换一行即可)
storage: IStorageEngine = RemoteStorage(grpc_endpoint="lingjing:8772")

# 混合模式(热数据本地+冷数据远程)
storage: IStorageEngine = TieredStorage(
    hot=SQLiteStorage(),       # L0-L2 本地
    warm=PostgreSQLStorage(),  # L3-L4 远程
    cold=S3Storage()           # L5 归档
)
```

### 0.5 类比定位

| 系统 | 内核 | 分布式层 | 天机对应 |
|------|------|----------|----------|
| Linux | 单机内核 | Kubernetes | 天机内核 + 灵境 |
| SQLite | 嵌入式DB | PostgreSQL | 天机SQLite + 灵境分片 |
| Redis | 单线程 | Redis Cluster | 天机单进程 + 灵境集群 |
| Docker | 单机容器 | Swarm/K8s | 天机ONEDIR + 灵境Compose |

### 0.6 灵境状态：混沌

```
灵境 = 分布式编排层
状态: 混沌（未规划）
说明: 灵境的规划属于第二阶段任务，当前不涉及
预留: 天机通过Protocol接口为灵境预留扩展点
      - IStorageEngine → RemoteStorage
      - ISearchStrategy → DistributedSearch
      - IEventBus → DistributedEventBus

```

## 一、四个灵魂级拷问

### 拷问1：天机的产品定位

**天机 = AI通用记忆操作系统 (Universal AI Memory Operating System)**

| 维度 | 定位 |
|------|------|
| 不是什么 | 不是Mem0(记忆中间件)、不是Zep(时序图)、不是Letta(Agent运行时) |
| 是什么 | **AI记忆的操作系统层**——像Linux是计算OS一样，天机是AI记忆的OS |
| 一句话 | 天机让任何AI平台拥有持久、结构化、可进化的记忆能力 |
| 市场类比 | Linux:计算资源 :: Android:移动设备 :: **天机:AI记忆** |
| 核心壁垒 | ICME六层认知架构 + 内置LLM认知引擎 + L-Asset知识资产统一 |

**市场验证**：AI记忆市场2025年62.7亿美元→2030年284.5亿美元(CAGR 35%)。知识工作者每天浪费45分钟重建上下文=180+小时/年。通用记忆层是刚需。

### 拷问2："六层记忆+L-Asset"架构定位

**"六层记忆+L-Asset" = 天机核心聚阵(L3) = 产品核心差异化壁垒**

| 层级 | 名称 | 类比 | 独特价值 |
|------|------|------|----------|
| L0 Sensory | 感枢 | 感官神经 | 3秒实时捕获，其他系统无此层 |
| L1 Working | 运枢 | 工作记忆 | 会话上下文，与Sensory实时联动 |
| L2 Short-Term | 近枢 | 短期记忆 | 跨会话保持，72小时关键信息 |
| L3 Episodic | 忆枢 | 情景记忆 | 决策记录/AI经验，因果对存储 |
| L4 Semantic | 知枢 | 语义记忆 | 知识图谱/概念库，跨域知识融合 |
| L5 Meta | 元枢 | 元认知 | 策略自优化，系统自我进化 |

**L-Asset绑定协议**: `memory_id ↔ asset_id ↔ content_hash` 三重绑定

**与竞品对比**：
- Mem0: 向量优先，2层(短期+长期) → 天机6层+资产绑定
- Zep: 时序图，1层(事实+时间窗) → 天机6层+认知晋升
- Letta: Agent自管，2层(核心+归档) → 天机6层+门禁+进化
- **天机独有**: 感枢3秒捕获 + 元枢自进化 + L-Asset知识资产

### 拷问3：README.md索引系统定位

**README.md = 天机知识资产的目录索引层 = L-Asset的入口门户**

### 拷问4：DeepSeek内置到天机核心的差异化竞争定位

**DeepSeek = 天机的认知引擎 = "功能+事件"双驱动核心**

| 维度 | DeepSeek在天机中的定位 |
|------|----------------------|
| 不是什么 | 不是外部API调用、不是可选插件 |
| 是什么 | **天机认知引擎的内燃机**——记忆的写入/检索/进化/门禁全部由DeepSeek驱动 |
| 功能驱动 | 三循环驾驶(快速反应<100ms / 深度思考5min / 进化反思1天) |
| 事件驱动 | 因果对记录→进化信号→法则治理→架构演化 |
| 差异化 | 竞品用LLM做"事后摘要"，天机用DeepSeek做"事中认知"——记忆写入即认知 |

**市场差异化矩阵**：

| 竞品 | LLM角色 | 记忆层次 | 认知深度 | 知识资产 |
|------|---------|----------|----------|----------|
| Mem0 | 事后摘要 | 2层 | 浅(向量匹配) | 无 |
| Zep | 事后提取 | 1层+时间窗 | 中(时序推理) | 无 |
| Letta | Agent自管 | 2层 | 中(主动编辑) | 无 |
| **天机** | **内置认知引擎** | **6层** | **深(三循环驾驶)** | **L-Asset三重绑定** |

**DeepSeek Engram启示**：75%动态推理+25%静态查找的最优分配比。天机v10.0.1应借鉴此比例，将DeepSeek算力分配为：75%用于记忆认知(门禁/进化/推理)，25%用于静态检索(向量/关键词)。

## 四、SSS级指令体系——模块逐个跑通

### 4.1 指令体系设计原则

**每个模块必须独立跑通才能进入下一个模块**——这是铁律。

```
模块跑通标准(SSS级):
S1: Import通过 — import module无报错
S2: 类型检查通过 — mypy --strict无报错
S3: 单元测试通过 — pytest覆盖率≥80%
S4: 集成测试通过 — 与上下游模块联调通过
S5: 性能达标 — P99<模块SLA阈值
```

### 4.2 Phase 0 模块跑通指令

#### 模块Ω-03: protocols.py（最高优先，所有模块依赖）

```
指令P0-01: 创建protocols.py
  操作: 创建 core/shared/protocols.py
```

```python
class IEventBus(Protocol):
    def publish(event: DomainEvent) -> None: ...

    def unsubscribe(event_type: str, handler: Callable) -> None: ...
```

## 5.5 TCL统一规范语言架构 (Tianji Canonical Language)

### 5.5.1 问题定义

天机v9.1有6个碎片化语言处理点，但缺少统一规范语言层：

| 现有机制 | 代码位置 | 处理能力 | 缺陷 |
|----------|----------|----------|------|
| `expand_query` | llm_bridge.py:146 | 检索时扩展同义词 | 只管检索，不管存储归一化 |
| `_normalize()` | knowledge_extractor.py:433 | 去空格/语气词/lowercase | 太浅，不做实体对齐 |
| `tianji_auto_tag` | MCP工具 | 自动打标签 | 标签无规范，不同AI打不同标签 |
| `tianji_classify` | MCP工具 | 智能分层 | 只分层级，不归一化术语 |
| `normalize_message` | message_gateway.py:425 | 统一消息格式 | 只管格式，不管内容语义 |
| FTS5全文搜索 | sqlite_store.py | 关键词匹配 | 无法处理语义等价 |

**核心问题**：用户/AI用不同表述描述同一概念时，天机无法识别为同一实体，导致存储冗余和检索遗漏。

## 5.8 天机v10.0.1 运营规划

### 5.8.1 监控指标体系

| 指标类别 | 关键指标 | 采集方式 | 告警阈值 |
|----------|----------|----------|----------|
| 可用性 | 服务healthy | GET /api/health 每30s | 连续3次失败→告警 |
| 延迟 | API P99延迟 | Prometheus histogram | >1.5s→L1降级 |
| 容量 | 各层使用率 | memory_stats() 每5min | >85%→自动固结 |
| 错误 | 5xx错误率 | 日志采集 | >5%→L2降级 |
| 记忆 | 写入拒绝率 | QualityGate统计 | >20%→调优门禁 |
| LLM | DeepSeek API可用率 | 调用成功/总数 | <95%→L3降级 |
| MCP | 工具调用成功率 | 每次调用记录 | <95%→告警 |

### 5.8.2 四级降级策略

```
L0 正常: 全功能运行, health=healthy
L1 缓存: P99>1.5s 或 错误率>5% → TCL仅快速匹配, 跳过LLM增强
L2 启发: 天机不可用>5min → Agent用本地知识库, TCL降级为关键词
L3 保守: 天机不可用>30min → 完全本地模式, 恢复后逐级回升

恢复规则: 严禁跳级, L3→L2(5min观察)→L1(5min观察)→L0
```

### 5.8.3 备份与恢复策略

| 备份类型 | 频率 | 保留策略 | 存储位置 |
|----------|------|----------|----------|
| 全量备份 | 每日03:00 | 保留7天 | data/backups/full/ |
| 增量备份 | 每小时 | 保留24h | data/backups/incr/ |
| TCL术语表 | 每次变更 | 保留90天 | data/backups/tcl/ |
| 配置快照 | 每次变更 | 保留30天 | data/backups/config/ |

### 5.8.4 容量规划

| 部署规模 | 日记忆量 | 月存储增量 | 推荐硬件 | 预估成本 |
|----------|----------|------------|----------|----------|
| 个人 | 1000条 | ~500MB | 4核/8GB/100GB SSD | 0(本地) |
| 小团队 | 10000条 | ~5GB | 8核/16GB/500GB SSD | ¥500/月 |
| 中团队 | 50000条 | ~25GB | 16核/32GB/1TB SSD | ¥2000/月 |
| 企业 | 200000条 | ~100GB | 32核/64GB/2TB NVMe | ¥8000/月 |

### 5.8.5 运维Runbook

```
日常巡检 (每日):
  □ GET /api/health → status=healthy
  □ memory_stats() → 各层使用率<85%
  □ 日志无ERROR级别异常
  □ DeepSeek API响应正常

周度维护 (每周):
  □ 全量备份验证(解压+查询)
  □ TCL术语表统计(canonical_count, alias_avg)
  □ 性能基准复测(P50/P99对比上周)
  □ 固结日志审查(晋升率/降级率)

月度审计 (每月):
  □ Agent调度统计(TVP日志审查)
  □ QualityGate门禁统计(PASS/REJECT比例)
  □ 容量趋势分析(预测下月容量需求)
  □ 安全扫描(依赖漏洞+密钥检测)

故障响应:
  级别S1(服务不可用): 15min内响应, 1h内恢复
  级别S2(性能退化): 30min内响应, 4h内恢复
  级别S3(功能异常): 2h内响应, 24h内修复
```

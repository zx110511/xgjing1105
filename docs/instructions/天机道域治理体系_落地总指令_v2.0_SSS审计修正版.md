## 第一章：落地战略决策

### 1.1 核心决策：落地路径选择

```
决策编号: DDGE-DEC-001
决策内容: 道域治理引擎(DDGE)采用"渐进式嵌入"落地策略
决策理由:
  1. 天机v9.1已有8个可复用模块(law_domain/governance_pipeline/evolution_engine等)
  2. 全量重写风险高，渐进嵌入可复用已有投资
  3. 每个Phase独立可验证，失败可回滚
  4. 符合三才治理模型自身的"法则→度量→演化"闭环哲学

替代方案(已否决):
  A. 全量重写 — 否决理由: 风险高、周期长、已有投资浪费
  B. 仅文档化 — 否决理由: 不落地=未完成，违反"操作必记录"原则
  C. 外部独立系统 — 否决理由: 与天机核心脱节，无法形成治理闭环
```

### 1.3 落地总目标

```
DDGE完全落地的6个验收标准:
  G1: 三柱可运行 — 法则柱/度量柱/演化柱可独立启动并响应查询
  G2: 治理闭环通 — 法则→度量→演化→法则的闭环事件可端到端传递
  G3: 地煞可度量 — 36地煞每个都有8维度量分数和趋势数据
  G4: 差异可诊断 — 系统可自动诊断法则差距/度量差距/演化差距/一致性差距
  G5: 演化可执行 — L1参数调优可自动执行，L2/L3可半自动/人工审核
  G6: API可调用 — 16个DDGE REST端点全部可调用并返回正确数据
```

      def govern(self, context) -> GovernanceResult:
          if self._degraded_mode:
              return GovernanceResult(status="degraded", message="降级模式运行")
          law_state = self.law_pillar.get_current_state()
          metric_state = self.metric_pillar.measure(law_state)
          gap_report = self.evolve_pillar.diagnose(law_state, metric_state)
          evolve_plan = self.evolve_pillar.plan(gap_report)

              gap_report=gap_report,
              evolve_plan=evolve_plan
          )

      def execute_evolution(self, step: EvolveStep) -> EvolveResult:
          snapshot = self._create_snapshot()
          if snapshot is None:
              return EvolveResult(status="failed", reason="快照创建失败")
          try:
              result = self.evolve_pillar.execute(step)
          except Exception as e:
              self._rollback(snapshot)
              return EvolveResult(status="failed", reason=f"执行异常: {e}")
          verification = self.metric_pillar.verify_evolution(step, result)
          if not verification.passed:
              rollback_ok = self._rollback(snapshot)
              if not rollback_ok:
                  self._enter_safe_mode()
              return EvolveResult(status="rolled_back", reason=verification.reason)
          return result

## 第三章：全局回滚策略 (v2.0新增)

### 3.1 Phase级回滚

```
═══════════════════════════════════════════════════════════════
  Phase级回滚策略 — 每个Phase独立可回滚
═══════════════════════════════════════════════════════════════

触发条件: Phase验收标准未通过，且修复成本>重做成本

回滚步骤:
  Step 1: 停止当前Phase所有运行时组件
  Step 2: 恢复该Phase开始前的代码快照(git revert)
  Step 3: 验证回滚后系统可正常运行
  Step 4: 记录回滚原因到天机L3 Episodic层
  Step 5: 重新规划该Phase的执行方案

Phase间依赖保障:
  - Phase 0: 无依赖，可独立回滚
  - Phase 1: 依赖P0，回滚P1不影响P0
  - Phase 2: 依赖P1，回滚P2需验证P1法则柱不受影响
  - Phase 3: 依赖P2，回滚P3需验证P2度量柱不受影响
  - Phase 4: 依赖P1+P2+P3，回滚P4需验证三柱独立运行正常
  - Phase 5: 依赖P4，回滚P5仅删除路由文件
  - Phase 6: 验收阶段，回滚=重新验收

═══════════════════════════════════════════════════════════════
```

### 3.2 任务级回滚

```
触发条件: 单个任务执行失败

回滚策略:
  - P0-T1~T3(归档): 回滚=从archive/移回core/
  - P1-T1~T4(代码): 回滚=git revert该文件
  - P1-T5/T6(法则数据): 回滚=deprecate_law + 从L5删除
  - P2-T5(基线数据): 回滚=删除L3中的基线记录
  - P3-T3(演化执行): 回滚=执行EvoExecutor._rollback(snapshot)
  - P4-T5(容器集成): 回滚=从TianjiContainer注销DDGE模块
```

### 3.3 系统级回滚

```
触发条件: DDGE集成导致天机核心功能异常

回滚策略:
  Step 1: 从TianjiContainer注销DDGE模块(核心功能恢复)
  Step 2: 删除core/ddge/目录(彻底移除)
  Step 3: 删除ddge_routes.py和governance_routes扩展
  Step 4: 重启天机服务
  Step 5: 验证/api/health返回healthy(无DDGE字段)

保障: DDGE作为可选模块设计，移除后核心功能不受影响
```

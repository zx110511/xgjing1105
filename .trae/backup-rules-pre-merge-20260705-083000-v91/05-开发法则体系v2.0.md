## 二、宏观层 — 宪法级法则（系统根基）

### 第一法则: 杜绝乱码

**来源**: 2026-06-17 用户明确制定
**记忆锚点**: project_memory.md — "Split files must use UTF-8-SIG encoding — prevents garbled characters (第一法则)"
**推演依据**: 2026-06-17/18 共49个文件BOM污染事件

**定义**: 所有文件必须使用UTF-8-SIG编码，每个文件只能有一个UTF-8-SIG BOM

**执行标准**:
- Python文件: `# -*- coding: utf-8-sig -*-` + 单BOM
- TypeScript文件: UTF-8-SIG编码 + 单BOM
- 检测方法: `codecs.BOM_UTF8` 检测，多BOM必须清理
- 禁止: 多BOM叠加 / GBK编码 / Latin-1编码

**违规后果**: P0违规，立即修复

---

### 第二法则: 固化增强

**来源**: 2026-06-17 用户明确制定
**记忆锚点**: project_memory.md — "All 100% audited content must be enhanced and固化 — per 第二法则"
**推演依据**: 2026-06-18 多次修复后未固化导致状态回退

**定义**: 所有100%审计通过的内容必须增强和固化，禁止仅修复不固化

**执行标准**:
- 修复后必须写入天机L3 Episodic层
- 系统变更必须写入L5 Meta层
- 修复模式必须沉淀到L4 Semantic层
- 后期扩展: 不新建文件，仅编辑现有文件（避免路径问题）

**违规后果**: P0违规，修复视为未完成

---

### 灵魂拷问: 真的能用吗

**来源**: 03-质量与操作铁律.md 已有定义
**推演依据**: 2026-06-18 审计100%通过但页面崩溃事件

**定义**: "真的能用吗？" — 不是"代码写了"，而是"代码真的能跑通、能闭环、能支撑真实场景"

**执行标准**:
- 六维验证: 可实现性/真实场景/并发稳定/跨模块集成/兼容性/边界条件
- 评分门槛: >=9.95分（满分10分）
- 必须使用真实运行数据验证，禁止仅代码审查
- 审计必须基于天机v9.1后台运行产生的真实数据

**违规后果**: P0违规，审计视为无效

## 三、中观层 — 开发规范法则（开发流程）

### 第三法则: API响应解包

**来源**: 2026-06-18/19 推演（3次重复崩溃）
**推演依据**:
- 2026-06-18: 知识图谱 `Cannot read 'slice'` — AxiosResponse未解包
- 2026-06-18: SSS审计 `Cannot read 'nodes'` — 同一问题
- 2026-06-19: SSS审计 `Cannot read 'filter'` — 同一问题第三次出现

**定义**: 所有axios响应必须显式解包`.data`，使用`response?.data ?? response`模式

**执行标准**:
```typescript
// ✅ 正确
const data = response?.data ?? response

// ❌ 错误（导致崩溃）
const data = response  // 存储了整个AxiosResponse对象
```

**违规后果**: P1违规，导致前端崩溃

---

### 第四法则: 降级完整性

**来源**: 2026-06-18 推演（1次严重事件）
**推演依据**: `main_status.py`降级分支仅读取`container._modules`(16个)，遗漏`_FULL_MODULE_CATALOG`(52个)，导致36个模块显示"未安装"

**定义**: 降级分支必须覆盖所有数据源，不能仅读取部分数据

**执行标准**:
- 降级逻辑必须显式枚举所有数据源
- 降级返回的数据必须包含完整的数据集
- 降级分支必须与主分支返回相同的数据结构

**违规后果**: P1违规，导致功能缺失

---

### 第五法则: 空值防御

**来源**: 2026-06-18/19 推演（3次重复崩溃）
**推演依据**:
- `topology.nodes.slice()` — nodes为undefined
- `auditData.results.filter()` — results为undefined
- `metrics.T1_scale.nodes` — T1_scale为undefined

**定义**: 所有数组操作（slice/filter/map/forEach）前必须进行存在性检查

**执行标准**:
```typescript
// ✅ 正确
if (Array.isArray(data?.items)) {
  data.items.filter(...)
}

// ❌ 错误
data.items.filter(...)  // items可能为undefined
```

**违规后果**: P1违规，导致前端崩溃

---

### 第六法则: 数据契约对齐

**来源**: 2026-06-19 推演（1次严重事件）
**推演依据**: 后端返回`{ total_nodes, score, grade }`扁平格式，前端期望`{ results: [{ code, passed, weight }] }`嵌套格式

**定义**: 后端API返回格式必须与前端期望格式对齐，或前端必须实现数据适配器

**执行标准**:
- API变更时必须同步更新前端类型定义
- 后端返回扁平格式时，前端必须实现`normalizeXxxData()`适配器
- 数据契约文档必须与代码同步更新

**违规后果**: P1违规，导致数据无法渲染

---

### 第七法则: 热重载感知

**来源**: 2026-06-17/18/19 推演（3次重复重启需求）
**推演依据**:
- 2026-06-17: Python修改后需重启uvicorn
- 2026-06-18: 276个路由注册后需重启服务
- 2026-06-19: 循环延迟修复后需重启后端

**定义**: 后端Python修改不生效时，必须重启服务；前端TypeScript修改通过`npm run build`生效

**执行标准**:
- 每次后端Python修改后，必须明确告知用户需要重启服务
- 前端TypeScript修改后，必须执行`npm run build`验证
- 重启方法必须与桌面快捷方式启动方法对齐

**违规后果**: P2违规，导致修改不生效

---

### 第八法则: 审计阈值同步

**来源**: 2026-06-19 推演（1次严重事件）
**推演依据**: 审计脚本`module_count=(1,50)`与实际62个模块不匹配，导致0/0得分

**定义**: 审计阈值必须与实际数据范围同步，避免因阈值过时导致审计失效

**执行标准**:
- 数据规模变化时必须同步更新审计脚本的阈值参数
- 审计脚本必须包含阈值自动检测机制
- 审计结果为0/0时，必须检查阈值是否过时

**违规后果**: P1违规，导致审计失效

---

### 第九法则: BOM单例

**来源**: 2026-06-17/18 推演（2次重复事件）
**推演依据**:
- 2026-06-17: 44个文件多BOM污染
- 2026-06-18: 5个文件BOM残留

**定义**: 每个文件只能有一个UTF-8-SIG BOM，禁止多BOM叠加

**执行标准**:
- 使用`codecs.BOM_UTF8`检测多BOM
- 多BOM文件必须清理为单BOM
- 文件写入前必须检查BOM数量

**违规后果**: P1违规，导致乱码

## 四、微观层 — 实现细节法则（实现细节）

### 第十法则: 资源边界

**来源**: 2026-06-16 推演（1次严重事件）
**推演依据**: TCP连接泄漏、CausalPair 2000条积累、Container Core缓存无限增长、60+后台循环同时运行导致CPU 100%

**定义**: 所有资源（连接/缓存/队列/循环）必须设置上限，禁止无限增长

**执行标准**:
- TCP连接池: 最大5连接/Server
- 缓存: FIFO淘汰，最大100条
- 队列: 最大2000条
- 后台循环: 最大并发3/Agent，系统总并发20
- 巩固循环: 60秒冷却

**违规后果**: P0违规，导致系统资源耗尽

---

### 第十一法则: 路径唯一性

**来源**: 2026-06-05 推演（1次严重事件）
**推演依据**: 僵尸数据库（data\icme.db 11.7MB仅1条记忆）、嵌套路径错误（data\data\.memory\）、模块绕过MEMORY_DATA_PATH

**定义**: 所有路径必须从唯一权威源获取，禁止硬编码路径

**执行标准**:
- Python路径: `from core.config import get_python_executable` / `sys.executable` fallback
- 数据路径: 从`MEMORY_DATA_PATH`环境变量获取
- 禁止: 硬编码绝对路径 / 嵌套路径 / 僵尸文件

**违规后果**: P1违规，导致数据丢失

---

### 第十二法则: Hook闭环

**来源**: 2026-06-05 推演（1次严重事件）
**推演依据**: 对话捕获hook未集成，数据库有完整表结构但0条实际捕获记录

**定义**: 所有hook必须形成闭环，禁止有hook注册但无实际执行

**执行标准**:
- hook注册后必须验证实际执行
- 对话捕获hook必须在IDE中集成
- 捕获端点必须返回200 OK状态
- 捕获统计必须解析JSON字符串为字典

**违规后果**: P1违规，导致数据丢失

---

### 第十三法则: 审计基准真实

**来源**: 2026-06-05 推演（1次严重事件）
**推演依据**: 审计报告混淆Python源文件数与容器注册模块数，将53个库文件误判为未集成

**定义**: 审计基准必须基于真实运行数据，禁止基于静态代码分析

**执行标准**:
- 审计必须使用天机v9.1后台运行产生的真实数据
- 审计必须区分源文件数与容器注册模块数
- 审计必须验证实际功能闭环，而非仅代码存在
- 审计结果必须可复现

**违规后果**: P0违规，导致审计无效

---

### 第十四法则: 拼写精确

**来源**: 2026-06-18 推演（1次严重事件）
**推演依据**: `pend_activ`拼写错误（应为`pend_active`），导致模块状态判定失败

**定义**: 所有状态字符串、枚举值、配置键名必须精确匹配，禁止拼写错误

**执行标准**:
- 状态字符串必须与前端判定逻辑完全匹配
- 枚举值必须与后端定义完全一致
- 配置键名必须与文档完全一致
- 修改状态字符串时必须全链路搜索替换

**违规后果**: P1违规，导致功能失效

---

### 第十五法则: 后台循环限流

**来源**: 2026-06-16 推演（1次严重事件）
**推演依据**: 60+后台循环同时运行导致CPU 100%

**定义**: 后台循环必须设置限流机制，禁止无限并发

**执行标准**:
- 每个Agent最大并发3个循环
- 系统总并发不超过20个循环
- 循环间隔必须>=1秒
- 循环必须设置超时机制

**违规后果**: P0违规，导致系统资源耗尽

## 五、微微观层 — 防御性法则（防御编程）

### 第十六法则: 类型断言

**来源**: 2026-06-19 推演
**推演依据**: 多次因类型不匹配导致的运行时错误

**定义**: 所有跨层边界的数据传递必须进行类型断言

**执行标准**:
- API响应必须验证字段类型
- 函数参数必须标注类型
- 返回值必须标注类型
- 禁止使用`any`类型

**违规后果**: P2违规，导致运行时错误

---

### 第十七法则: 异常传播

**来源**: 2026-06-18 推演
**推演依据**: 多次因异常被静默吞没导致问题难以定位

**定义**: 异常必须向上传播或显式处理，禁止静默吞没

**执行标准**:
- `try/except`块必须有`except`处理逻辑
- 禁止空`except: pass`
- 异常必须记录到天机L3 Episodic层
- 异常必须包含上下文信息

**违规后果**: P2违规，导致问题难以定位

---

### 第十八法则: 并发安全

**来源**: 2026-06-16 推演
**推演依据**: TCP连接泄漏和缓存竞态条件

**定义**: 共享资源访问必须线程安全，禁止竞态条件

**执行标准**:
- 共享变量必须使用锁保护
- 连接池必须线程安全
- 缓存必须使用线程安全容器
- 禁止跨线程共享可变状态

**违规后果**: P1违规，导致数据不一致

---

### 第十九法则: 幂等性保障

**来源**: 2026-06-18 推演
**推演依据**: 多次重复修复同一问题，说明修复不具备幂等性

**定义**: 修复操作必须幂等，重复执行不产生副作用

**执行标准**:
- 修复脚本必须可重复执行
- 数据初始化必须幂等
- 状态更新必须基于当前状态判断
- 禁止: 修复A导致B回退

**违规后果**: P2违规，导致状态回退

---

### 第二十法则: 变更可追溯

**来源**: 2026-06-18 推演
**推演依据**: 多次修复后无法追溯变更历史

**定义**: 所有代码变更必须可追溯，禁止无记录变更

**执行标准**:
- 变更必须标注`[FIX-xxx]`标签
- 变更必须记录到天机L3 Episodic层
- 变更必须更新README变更日志
- 变更必须通过灵魂拷问验证

**违规后果**: P2违规，导致变更不可追溯

## 七、经验复用机制 (新增)

### 经验库架构

**经验库位置**: L4 Semantic层

**经验分类体系**:
```
经验库 (L4 Semantic)
├── 开发经验 (dev_exp)
│   ├── 修复模式 (fix_pattern) — 第X法则对应的修复模式
│   ├── 架构模式 (arch_pattern) — 模块架构经验
│   └── 性能模式 (perf_pattern) — 性能优化经验
├── 运维经验 (ops_exp)
│   ├── 部署模式 (deploy_pattern) — 部署流程经验
│   ├── 监控模式 (monitor_pattern) — 监控告警经验
│   └── 故障模式 (fault_pattern) — 故障处理经验
└── 业务经验 (biz_exp)
    ├── 创作模式 (create_pattern) — 小说创作经验
    ├── 审校模式 (review_pattern) — 审校流程经验
    └── 协作模式 (collab_pattern) — Agent协作经验
```

### 经验复用流程

```
Step 1: 识别 → 识别当前任务类型，匹配经验类别
Step 2: 检索 → memory_recall查询L4 Semantic层相关经验
Step 3: 验证 → 验证经验适用性（场景匹配度>=80%）
Step 4: 应用 → 应用经验模式，避免重复踩坑
Step 5: 沉淀 → 新经验写入L4 Semantic层
Step 6: 演化 → 经验模式持续优化（每30天review）
```

### 经验复用强制规则

**规则1: 经验优先查询**
- 非trivial任务必须先查询L4 Semantic层经验
- 豁免: 仅限简单配置选择 (complexity=low)

**规则2: 经验应用验证**
- 应用经验前必须验证场景匹配度>=80%
- 匹配度<80%时必须调整经验模式

**规则3: 经验沉淀强制**
- 新问题解决后必须沉淀经验到L4 Semantic层
- 经验必须包含: 问题描述 + 解决方案 + 适用场景 + 避坑指南

**规则4: 经验演化周期**
- 每30天review一次经验库
- 连续90天未使用的经验标记为"待退役"
- 连续180天未使用的经验标记为"退役"

### 经验复用示例

**示例1: 修复模式复用**
```

应用: 使用`response?.data ?? response`模式
```

## 十、法则演进机制

1. **问题识别**: 同一问题出现2次以上 → 推演新法则
2. **法则草案**: 记录到L3 Episodic层
3. **法则验证**: 经历至少3次实际应用验证
4. **法则固化**: 写入本文件 + project_memory.md
5. **法则退役**: 连续30天无违反 → 标记为稳定法则

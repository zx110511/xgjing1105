## 二、TVP透明调度协议 (强壮)

### Agent切换声明 (强制)

每次Agent切换必须声明:
```
[TVP] 🎯 Agent切换:
├─ 当前: @{current} → 目标: @{target}
├─ 任务: {task_type}
├─ 上下文: {50字摘要}
├─ 状态: {planning/executing/reviewing/completed}
└─ 数据流: {upstream→current→downstream}
```

### 技能调用声明 (新增)

每次技能调用必须声明:
```
[TVP-SKILL] 🔧 技能调用:
├─ 调用者: @{agent}
├─ 技能: {skill_name}
├─ 目标Agent: @{skill_owner}
├─ 参数摘要: {50字}
└─ 预期结果: {output_type}
```

### 记忆操作声明 (新增)

每次跨层记忆操作必须声明:
```
[TVP-MEM] 🗃️ 记忆操作:
├─ 操作者: @{agent}
├─ 操作: {remember/recall/consolidate}
├─ 层级: {L0-L5}
├─ 内容摘要: {50字}
└─ 关联Agent: {affected_agents}
```

**违规**: 未声明切换 → P1违规 | 连续3次 → 信任降级 | 影响系统稳定性 → P0违规

### 调度可视化与可追溯 (新增·强壮)

#### 触发场景
- 复杂任务涉及3个以上Agent协作
- 需要向用户展示决策过程与Agent分工
- 事后审计与复盘

#### 执行流程
1. **调度计划生成**
   - 解析任务依赖图（顺序/并行）
   - 生成甘特图式调度计划
   - 标记关键路径与风险点
2. **实时追踪**
   - 每个Agent开始/完成时输出TVP事件
   - 收集执行耗时、工具调用次数、记忆读写次数
   - 检测异常（超时、错误、重复调用）
3. **可视化输出**
   ```
   [TVP-Timeline] 任务: proj-001
   06:14:23 🎯@天枢(tianshu)  START  plan
   06:14:25 🔎@洞察(dongcha)  SWITCH 意图分析 (45ms)
   06:14:28 🎯@天枢(tianshu)  SWITCH 方案评估 (12ms)
   06:14:30 ✍️@妙笔(miaobi)   SWITCH 内容创作 (2.3s)
   06:14:33 🔍@明镜(mingjing) SWITCH 质量审校 (1.1s)
   06:14:35 🎯@天枢(tianshu)  END    总耗时 12s
   ```
4. **归档**
   - 完整调度记录存入天机 L3 Episodic
   - 生成调度效率报告（Agent利用率、等待时间、瓶颈识别）

#### 格式规范
- TVP格式：`[TVP] {emoji}@{name}({id}) → {action} | {detail}`
- 时间戳精确到毫秒
- 必须包含：Agent标识、动作类型、耗时、状态
- 违规标记：未声明切换高亮为 🔴

#### 主责Agent
@tianshu (调度总控) | @qianli (监控数据)

## 三、六大协作模式 (强壮)

| 模式 | 适用场景 | 核心特征 | Agent归属 |
|------|---------|---------|----------|
| A-串行 A→B→C→D | 顺序依赖 | 质量可控 | @miaobi→@mingjing→@jinshu |
| B-并行 →[A,B,C]→聚合 | 独立分析 | 速度快 | [@tiansuan+@qianli+@zhenshan] |
| C-层级 主控→子协调→工作者 | 大型分治 | 可扩展 | @tianshu→@jingwei→[@miaobi×N] |
| D-工业 S0→S1→...→S6 | 生产级交付 | 可追溯 | 见工业化生产流程 |
| E-事件驱动 发布者→[事件总线]→订阅者 | 松耦合异步 | 高可用 | @qianli→EventBus→@zhuiguang |
| F-进化闭环 检测→分析→改进→验证→沉淀 | 系统自改进 | 自进化 | @qianli→@zhuiguang→@huasheng→@tiewei→@yiku |

### 快速选择决策树

```
有严格顺序? → 是 → 生产级? → 是 → 模式D / 否 → 模式A
               → 否 → 完全独立? → 是 → 模式B / 否 → 需要异步? → 是 → 模式E
                                                        → 否 → 规模大? → 模式C / 否 → 自改进? → 模式F / 否 → 模式A
```

## 十二、MCP技能必然使用规则（新增·强制执行）

### 12.1 必然使用判定矩阵（铁律）

| 任务类型 | MCP工具 | 必然使用条件 | 豁免条件 | 违规等级 |
|---------|---------|-------------|---------|---------|
| **非平凡决策** | `memory_recall` | complexity≠low | 简单配置选择 | P0 |
| **写操作** | `memory_remember` | 任何写操作 | 仅读取操作 | P0 |
| **多Agent任务** | `agent_dispatch` | 涉及≥2个Agent | 单Agent任务 | P1 |
| **规则合规验证** | `rule_evaluate` | 架构/规则变更 | 无 | P1 |
| **意图解析** | `context_extract` | 复杂多意图输入 | 单一明确指令 | P2 |
| **故障追溯** | `memory_recall` + 反思环 | 系统异常 | 无 | P0 |
| **资源搜索** | WebSearch | 推演深度≥6 | 推演深度<4 | P1 |
| **系统命令** | `execute_command` | 需系统级操作 | 内置工具可完成 | P2 |

### 12.2 必然使用执行流程（铁律）

```
Step 1: 任务复杂度评估
  ├── @tianshu评估complexity (low/medium/high/critical)
  ├── complexity≠low → 进入Step 2
  └── complexity=low → 可豁免MCP调用（但建议记录）

Step 2: MCP工具匹配
  ├── 根据任务类型匹配必然使用的MCP工具
  ├── 生成"必然使用清单"
  └── 输出: MCP_USE_REQUIRED = [tool1, tool2, ...]

Step 3: 执行前声明（TVP-MEM/TVP-SKILL）
  ├── 每个MCP工具调用前必须声明
  ├── 声明格式见第二节TVP协议
  └── 输出: TVP声明

Step 4: MCP工具执行
  ├── 按声明顺序执行MCP工具
  ├── 每个工具执行结果必须记录
  └── 输出: 执行结果

Step 5: 执行后验证
  ├── 验证MCP_USE_REQUIRED清单是否全部执行
  ├── 未执行的MCP工具需说明原因
  └── 输出: MCP_USE_VERIFIED = true/false

Step 6: 违规处理
  ├── MCP_USE_VERIFIED=false → 记录P0/P1违规
  ├── 连续3次违规 → 信任降级
  └── 输出: 违规报告
```

### 12.3 MCP工具调用强制约束

**强制调用场景**:
1. **任何非平凡决策前**: 必须调用`memory_recall`检索历史决策
2. **任何写操作后**: 必须调用`memory_remember`记录到对应层级
3. **多Agent协作前**: 必须调用`agent_dispatch`获取调度推荐
4. **架构/规则变更前**: 必须调用`rule_evaluate`验证合规性
5. **复杂意图解析**: 必须调用`context_extract`结构化解析
6. **故障发生后**: 必须调用`memory_recall`+反思环追溯根因
7. **推演深度≥6**: 必须调用WebSearch搜索外部资源

**禁止行为**:
- 禁止跳过MCP工具直接使用内置工具完成MCP必用任务
- 禁止伪造MCP工具调用记录
- 禁止在MCP工具不可用时静默降级（必须TVP声明）
- 禁止连续3次以上MCP工具调用失败后不升级

### 12.4 MCP工具降级处理协议

当MCP工具不可用时，必须按以下协议处理：

```
MCP工具不可用
  ↓
Step 1: TVP声明降级
  └── [TVP-MEM] 🗃️ 记忆操作降级:
      ├── 操作者: @{agent}
      ├── 原计划: {MCP工具}
      ├── 降级原因: {不可用原因}
      ├── 降级方案: {替代方案}
      └── 数据暂存: {暂存位置}

Step 2: 离线队列暂存
  ├── 写操作 → 暂存到 .tianji/offline_writes.json
  ├── 读操作 → 标记为"待检索"，后续补检
  └── 调度操作 → 降级为内置工具+人工编排

Step 3: 恢复后补执行
  ├── MCP恢复后自动执行离线队列
  ├── 补执行的记录标记为"延迟写入"
  └── 输出: 补执行报告
```

## 十三、智能调度必然执行规则（新增·强制执行）

### 13.1 智能调度触发条件（铁律）

| 触发条件 | 调度模式 | 必然执行 | 豁免 |
|---------|---------|---------|------|
| **任务涉及≥2个Agent** | B/C/D/E/F | `agent_dispatch` | 无 |
| **任务复杂度≥medium** | A/C/D | `agent_dispatch`+`memory_recall` | 无 |
| **任务优先级≥high** | A/C/D | `agent_dispatch`+`rule_evaluate` | 无 |
| **架构决策/技术选型** | C层级 | `agent_dispatch`+`memory_recall`+`rule_evaluate` | 无 |
| **系统级操作** | C层级 | `agent_dispatch`+`rule_evaluate` | 无 |
| **故障追溯** | F进化闭环 | `agent_dispatch`+`memory_recall`+反思环 | 无 |
| **多里程碑项目** | D工业化 | `agent_dispatch`+`pipeline_create` | 无 |

### 13.2 智能调度执行流程（铁律）

```
Step 1: 任务特征分析
  ├── @tianshu分析任务特征(类型/复杂度/优先级/Agent数)
  ├── 匹配调度触发条件
  └── 输出: DISPATCH_REQUIRED = true/false

Step 2: 调度推荐获取
  ├── DISPATCH_REQUIRED=true → 调用agent_dispatch
  ├── 获取推荐Agent列表+置信度+可用性
  └── 输出: 推荐Agent = [agent1, agent2, ...]

Step 3: 调度决策
  ├── 推荐Agent available=true → 采用推荐
  ├── 推荐Agent available=false → 降级为@tianshu直接执行
  ├── 推荐置信度<0.7 → 降级为@tianshu直接执行
  └── 输出: 最终Agent = {decision}

Step 4: TVP声明（强制）
  ├── 每次Agent切换必须TVP声明
  ├── 降级情况必须说明原因
  └── 输出: TVP声明

Step 5: 调度执行
  ├── 按决策结果执行任务
  ├── 实时监控执行状态
  └── 输出: 执行结果

Step 6: 调度后验证
  ├── 验证DISPATCH_REQUIRED是否满足
  ├── 验证TVP声明是否完整
  └── 输出: DISPATCH_VERIFIED = true/false
```

### 13.3 智能调度强制约束

**强制调度场景**:
1. **多Agent协作**: 必须通过`agent_dispatch`获取推荐
2. **复杂任务**: 必须通过`agent_dispatch`+`memory_recall`决策
3. **高优先级任务**: 必须通过`agent_dispatch`+`rule_evaluate`验证
4. **架构决策**: 必须通过三工具联合（dispatch+recall+evaluate）
5. **系统级操作**: 必须通过`agent_dispatch`+`rule_evaluate`
6. **故障追溯**: 必须通过完整六步决策流水线
7. **多里程碑**: 必须通过`agent_dispatch`+`pipeline_create`

**禁止行为**:
- 禁止跳过`agent_dispatch`直接人工指定Agent
- 禁止忽略`agent_dispatch`推荐结果
- 禁止降级时不TVP声明
- 禁止连续3次调度失败不升级

### 13.4 调度失败处理协议

```
调度失败
  ↓
Step 1: 失败原因分析
  ├── MCP不可用 → 启用MCP降级协议(12.4)
  ├── 推荐Agent不可用 → 降级为@tianshu直接执行
  ├── 推荐置信度低 → 降级为@tianshu直接执行
  └── 任务超时 → 升级到@tianji

Step 2: TVP声明降级
  └── [TVP] 🎯 Agent切换降级:
      ├── 当前: @{current} → 目标: @{tianshu_direct}
      ├── 任务: {task_type}
      ├── 降级原因: {reason}
      └── 状态: degraded_executing

Step 3: 降级执行
  ├── @tianshu直接使用内置工具完成
  ├── 实时记录降级执行过程
  └── 输出: 降级执行结果

Step 4: 恢复后补执行
  ├── 系统恢复后评估是否需要补执行
  ├── 需要补执行 → 重新走调度流程
  └── 输出: 补执行报告
```

### 13.5 调度必然使用审计

每次任务结束后，必须进行调度必然使用审计：

| 审计项 | 审计标准 | 违规等级 |
|--------|---------|---------|
| **DISPATCH_REQUIRED判定** | 是否正确判定调度必要性 | P1 |
| **agent_dispatch调用** | 必要时是否调用 | P0 |
| **TVP声明完整** | 每次切换是否声明 | P1 |
| **降级原因记录** | 降级是否记录原因 | P2 |
| **记忆操作完整** | 是否调用memory_recall/remember | P0 |
| **规则验证执行** | 是否调用rule_evaluate | P1 |

# 06-常识类法则 (Common-Sense Laws) v2.0

**优先级**: P0-强制 | **激活**: 始终生效 | **适用**: 全Agent·每次对话

> **核心论断**: 天机系统已设计好的功能如果不在对话中使用，等于功能不存在。
> **执行原则**: 常识级功能必须自动触发，不触发即预警。

---

## 一、常识定义

**常识 = 每次对话都应自然使用的天机功能**。这些功能不是"可选"的，而是"默认必须执行的"。

| 常识类别 | 包含功能 | 最低调用率 | 未使用预警 |
|---------|---------|-----------|----------|
| 对话录入审计 | tianji_intercept / memory_remember / 对话自动审计 | 100% | ⚠️ P0致命 |
| 记忆检索 | memory_recall / tianji_semantic_search | >=80% | ⚠️ P1严重 |
| MCP技能调用 | Skill工具 / MCP工具调用 | >=60% | ⚠️ P0致命 |
| MCP规范使用 | TVP-MCP声明 / 参数验证 / 频率限制 | 100% | ⚠️ P0致命 |
| 智能体调度 | agent_dispatch / TVP声明 | 需协作时>=90% | ⚠️ P1严重 |
| 系统健康检查 | tianji_health / system_diagnose | 每次启动时100% | ⚠️ P1严重 |
| 记忆整合 | memory_consolidate / memory_stats | 每30分钟 | ⚠️ P2一般 |

---

## 二、常识强制规则

### 常识1: 对话全量录入审计 (P0-致命)

**规则**: 每次对话必须完成以下闭环：
```
对话开始 → tianji_intercept(注入上下文)
    ↓
对话进行 → 工具调用 + 代码生成
    ↓
对话结束 → memory_remember(结果+上下文) / 离线队列(降级时)
    ↓
          → memory_consolidate(整合优化)
    ↓
          → 对话自动审计(完整性检查)
```

**预警**: 任何一环缺失 → 立即发出P0警告："对话记录不完整，违反常识1"

**检查清单**:
- [ ] tianji_intercept 是否调用？
- [ ] 关键决策是否记录到L3 Episodic？
- [ ] 系统变更是否记录到L5 Meta？
- [ ] 对话结束前是否完成自动审计？

---

### 常识2: MCP技能调用率 (P0-致命)

**规则**: 每次对话中MCP技能(Skill工具)调用率必须>=60%

**调用率计算**: `(使用Skill的回复数 / 总回复数) × 100%`

**必须使用的Skill (每次对话)**:
| Skill | 触发时机 | 优先度 |
|-------|---------|--------|
| memory-recall | 决策前查询天机 | P0 |
| memory-remember | 操作后记录 | P0 |
| context-extract | 任务解析阶段 | P0 |
| agent-dispatch | 多Agent协作时 | P1 |
| skill-route | 用户指令路由 | P0 |
| dialogue-quality | 对话结束时 | P1 |
| system-diagnose | 系统异常时 | P1 |

**预警**: 对话超过3轮未调用任何MCP技能 → "⚠️ MCP调用率为0%，违反常识2，请立即使用天机记忆功能"

---

### 常识3: MCP规范使用 (P0-致命) — 新增

**规则**: 所有MCP工具调用必须遵循TVP-MCP声明规范

**TVP-MCP声明格式**:
```
[TVP-MCP] 工具: {tool_name} | Server: {server_name} | 参数: {key_params} | 目的: {purpose}
```

**MCP工具优先级矩阵**:
| 工具 | 用途 | 优先级 | 调用频率限制 |
|------|------|--------|------------|
| tianji_intercept | 对话拦截+上下文注入 | P0 | 每次对话开始 |
| memory_remember | 写入记忆 (指定层级) | P0 | 每次操作后 |
| memory_recall | 检索记忆 (语义/精确) | P0 | 每次决策前 |
| memory_forget | 软删除记忆 | P1 | 需删除时 |
| memory_consolidate | 记忆整合 | P1 | 每30分钟 |
| tianji_health | 系统健康检查 | P1 | 每次启动 |
| tianji_auto_tag | 自动标签 | P2 | 每次写入 |
| tianji_summarize | 智能摘要 | P2 | 大量数据时 |
| tianji_semantic_search | 语义搜索 | P2 | 复杂查询时 |

**MCP参数验证规范**:

**memory_remember参数验证**:
```
必填参数:
- content: string (非空，长度>=10字符)
- layer: sensory|working|short_term|episodic|semantic|meta

选填参数:
- tags: array (至少2个标签)
- priority: low|medium|high|critical
- metadata: object (可选元数据)

验证规则:
- content长度<10 → ⚠️ 警告：内容过短，建议补充
- layer未指定 → ❌ 错误：必须指定层级
- tags数量<2 → ⚠️ 警告：建议添加更多标签
```

**memory_recall参数验证**:
```
必填参数:
- query: string (非空，长度>=5字符)

选填参数:
- layers: array (可选层级列表)
- limit: integer (1-100，默认10)
- threshold: float (0.0-1.0，默认0.5)

验证规则:
- query长度<5 → ⚠️ 警告：查询过短，建议细化
- limit>100 → ❌ 错误：超出最大限制
- threshold<0.0 → ❌ 错误：阈值无效
```

**MCP调用频率限制**:
- 单Agent最大并发: 3个MCP调用
- 系统总并发: 20个MCP调用
- MCP连接池: 5连接/Server
- 调用间隔: >=1秒（防止高频调用）

**MCP错误分级响应**:
| 错误级别 | 响应时限 | 必须动作 |
|---------|---------|---------|
| Critical | <30s | 停止调用 + L3+L5写入 + 告警 + 降级 |
| Major | <2min | 重试/回退 + L3写入 + 通知用户 |
| Minor | <5min | 优雅降级 + L1/L3写入 |

**预警**: MCP调用未遵循TVP-MCP声明 → "⚠️ 违反常识3：MCP调用未声明，请使用TVP-MCP格式"

---

### 常识4: 记忆优先决策 (P0-致命)

**规则**: 非平凡决策必须先查询天机记忆

**决策级别判定**:
| 级别 | 特征 | 记忆查询要求 |
|------|------|------------|
| trivial | 单步/无依赖/无风险 | 可跳过 |
| standard | 多步/有依赖/低风险 | 必须查L4+部分L3 |
| critical | 多步/高风险/不可逆 | 必须查L4+L3+L5 |

**预警**: standard+级别决策未查记忆 → "⚠️ 违反常识4：闭门造车，请先查询天机记忆"

---

### 常识5: 智能体调度适用 (P1-严重)

**规则**: 满足以下任一条件时，必须调用agent_dispatch：
- 任务需要多Agent协作
- 任务复杂度>=standard
- 跨模块操作
- 需要TVP透明调度

**预警**: 满足条件但未调度 → "⚠️ 违反常识5：未使用智能体调度，TVP协议未执行"

---

### 常识6: 系统健康自检 (P1-严重)

**规则**: 每次对话开始时自动检查天机健康状态

```python
# 伪代码
if session_start or user_mentions_system:
    health = tianji_health()
    if not health.ok:
        warn("天机系统异常，进入降级模式")
```

**预警**: 启动时未健康检查 → "⚠️ 违反常识6：未知天机健康状态，存在降级风险"

---

### 常识7: 修复闭环审计 (P1-严重)

**规则**: 每次修复必须完成以下闭环：
```
定位根因 → 修复代码 → 构建验证 → 功能验证 → 写入记忆
```

**预警**: 任一环节缺失 → "⚠️ 违反常识7：修复未闭环，可能导致状态回退"

---

### 常识8: README同步 (P2-一般)

**规则**: 新建文件夹必须同时创建README.md，变更必须更新README

**预警**: 新建文件夹无README → "⚠️ 违反常识8：灵魂框架缺失"

## 三、对话中自动预警触发器

在每次回复前，天枢(@tianshu)必须执行以下常识自检：

```
┌─ 常识自检清单 (每轮对话) ─────────────────┐
│ □ 本轮是否调用了MCP技能？                  │
│   否 → ⚠️ 警告：MCP调用率为0%              │
│ □ 本轮MCP调用是否遵循TVP-MCP声明？         │
│   否 → ⚠️ 警告：MCP调用未声明              │
│ □ 本轮决策是否查询了天机记忆？              │
│   否且非trivial → ⚠️ 警告：闭门造车        │
│ □ 本轮是否完成了操作记录？                  │
│   修改了代码但未记录 → ⚠️ 警告：记录缺失    │
│ □ 是否需要进行Agent调度？                   │
│   需要但未TVP声明 → ⚠️ 警告：调度缺失      │
│ □ 天机健康状态是否正常？                    │
│   超过30分钟未检查 → ⚠️ 警告：健康未知     │
│ □ MCP调用频率是否超限？                     │
│   超限 → ⚠️ 警告：高频调用风险             │
└──────────────────────────────────────────┘
```

## 四、预警格式规范

所有常识违反预警必须使用以下格式：

```
═══════════════════════════════════════════
⚠️  天机常识预警 [#{常识编号}] — {违反级别}
═══════════════════════════════════════════
违反项: {具体违反的常识名称}
当前状态: {未执行的操作}
影响: {功能缺失/数据丢失/状态回退}
修复动作: {需要执行的具体操作}
═══════════════════════════════════════════
```

| 每次会话 | >=80% | 记忆检索率 |

## 六、MCP规范常识化检查表 (新增)

每次MCP调用必须完成以下检查：

### 检查项1: TVP-MCP声明
- [ ] 是否声明工具名称？
- [ ] 是否声明Server名称？
- [ ] 是否声明关键参数？
- [ ] 是否声明调用目的？

### 检查项2: 参数验证
- [ ] 必填参数是否完整？
- [ ] 参数类型是否正确？
- [ ] 参数范围是否合法？
- [ ] 参数长度是否合适？

### 检查项3: 频率限制
- [ ] 是否超过单Agent并发限制（3个）？
- [ ] 是否超过系统总并发限制（20个）？
- [ ] 是否超过连接池限制（5连接/Server）？
- [ ] 是否超过调用间隔限制（>=1秒）？

### 检查项4: 错误响应
- [ ] Critical错误是否<30s响应？
- [ ] Major错误是否<2min响应？
- [ ] Minor错误是否<5min响应？
- [ ] 错误是否记录到L3 Episodic？

## 七、常识演化机制

1. **新常识识别**: 功能设计完成且验证可用 → 自动纳入常识清单
2. **调用率统计**: 每10轮对话统计一次调用率
3. **阈值调整**: 根据实际使用情况调整最低调用率
4. **退役条件**: 连续90天无违规 → 标记为稳定常识

---

## 八、MCP规范演化机制 (新增)

1. **新工具识别**: MCP工具设计完成且验证可用 → 自动纳入优先级矩阵
2. **参数规范更新**: 参数变更时同步更新验证规范
3. **频率限制调整**: 根据系统负载调整频率限制
4. **错误响应优化**: 根据错误统计优化响应时限

# 04-操作法则 (Operations Laws) v4.0

**优先级**: P1-强制 | **激活**: 操作时 | **适用**: 全Agent

---

## 一、决策引擎: 六步流水线 (强壮)

```
Step 1: 识别 → 意图分类 + 复杂度评估 + 优先级判定
Step 2: 检索 → 天机L2-L5 + 联网资源 (破除闭门造车)
Step 3: 融合 → 去重 + 冲突检测 + 增强上下文
Step 4: 生成 → >=3方案 (critical级) / 基线方案 (standard级)
Step 5: 评估 → 灵魂拷问六维验证 + Stage Gate检查
Step 6: 记录 → L1 Working + L3 Episodic (重要) + L5 Meta (系统变更)
```

### 复杂度分级

| 级别 | 特征 | 决策路径 |
|------|------|---------|
| trivial | 单步/无依赖/无风险 | 直接执行，后记录 |
| standard | 多步/有依赖/低风险 | 完整6步流水线 |
| critical | 多步/高风险/不可逆 | 6步+人工确认+L5归档 |

### 豁免条件 (可跳过检索步骤)
- 简单配置选择 (complexity=low)
- 格式化/美化等纯机械操作
- 已有明确指令的执行操作

## 二、四级降级模型 (强壮)

```
Level 0: 正常模式 ──→ 天机v9.1全功能可用
Level 1: 缓存模式 ──→ 天机不可写，读本地缓存
Level 2: 启发模式 ──→ 天机完全不可用，规则驱动
Level 3: 保守模式 ──→ 仅允许安全操作，人工介入
```

### 降级触发条件

| 降级方向 | 触发条件 | 检测方式 |
|----------|---------|---------|
| L0→L1 | memory_remember超时>5s / 写入失败 | 工具返回异常 |
| L1→L2 | 连续3次读取失败 / 健康检查失败 | tianji_health |
| L2→L3 | 系统级异常 / 安全事件 | 多信号聚合 |

### 各级别操作规范

**Level 1: 缓存模式**
- 允许: 读取本地缓存 / 继续当前任务 / 离线队列暂存写操作
- 禁止: 依赖实时天机数据的决策 / 跨会话记忆引用
- 恢复: 天机恢复后回写离线队列 + 执行consolidate

**Level 2: 启发模式**
- 允许: 基于规则的操作 / 本地文件读写 / 预设模板
- 禁止: 任何天机调用 / 依赖记忆的决策 / 自动整合
- 恢复: 人工确认天机状态 → 逐步恢复 → 验证数据完整性

**Level 3: 保守模式**
- 允许: 只读操作 / 安全查询 / 人工指令执行
- 禁止: 所有写操作 / 自动化流程 / Agent间通信
- 恢复: 必须人工介入 → 系统级诊断 → 逐步解除限制

### 恢复流程 (严禁跳级)

```
L3→L2: 人工确认 + 稳定5min → L2
L2→L1: 天机恢复验证 + 稳定5min → L1
L1→L0: 离线队列同步 + consolidate + 稳定5min → L0
```

## 三、离线队列机制 (强壮)

### 队列位置
```
D:\元初系统\天机v10.0系列\.tianji\offline_writes.json
```

### 队列条目格式
```json
{
  "id": "offline-{NNN}",
  "timestamp": "ISO8601",
  "layer": "sensory|working|short_term|episodic|semantic|meta",
  "priority": "low|medium|high|critical",
  "tags": ["tag1", "tag2"],
  "content": "待写入的记忆内容"
}
```

### 操作规范
- **写入时机**: memory_remember失败时，自动将内容写入离线队列
- **同步时机**: 天机恢复后(tianji_health返回healthy)，批量同步离线队列
- **同步顺序**: 按priority排序 (critical > high > medium > low)
- **同步验证**: 每条写入后memory_recall验证，确认后从队列移除
- **容量保护**: 离线队列最大1000条，超出时critical优先保留

## 四、文件同步机制 (三道防线)

### 第一道防线: 启动扫描
- 触发: 每次对话启动时
- 方法: 遍历v10.0系列全部文件，计算SHA256哈希
- 比对: 与天机L4 Semantic中存储的文件哈希比对
- 差异处理: 新增→记录L4 / 修改→记录L3+更新L4 / 删除→记录L3+标记L4

### 第二道防线: 文件监控
- 触发: 文件变更事件
- 方法: 监听v10.0系列目录变更
- 处理: 实时同步到天机L4 Semantic

### 第三道防线: 定期对账
- 触发: 每30分钟
- 方法: 全量扫描+哈希比对
- 处理: 修复不一致+记录差异

## 五、对话记录机制 (双通道)

### 通道A: 对话内容 → ICME记忆层
- L0 Sensory: 原始输入缓存
- L1 Working: 当前会话上下文
- L3 Episodic: 重要决策和事件

### 通道B: 文件内容 → L4 Semantic
- 每次文件修改，内容摘要写入L4
- 双向锚定: conversation_id <-> file_path <-> memory_id

### 对话自动审计 (P0强制)
每次对话结束前必须审计:
1. 对话关键决策是否记录到天机
2. 修改的文件内容是否完整记录
3. 错误和修复是否记录
4. 待办事项是否更新
